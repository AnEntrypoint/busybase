BusyBase Docs

BusyBase is a minimal, self-hosted backend built on Bun and libSQL (SQLite). It provides a Supabase JS v2-compatible API, Ed25519 keypair authentication, and a pluggable hooks system — with zero config, no Docker, and no external database process.

Drop-in Supabase compatibility
BusyBase mirrors the Supabase JS client v2 API. Change your URL and most existing Supabase code will work unchanged.

Key facts

PropertyValue
RuntimeBun (TypeScript, zero transpile)
StoragelibSQL — SQLite-compatible, single file, no server process
AuthEd25519 keypair (WebCrypto) + email/password (bcrypt via Bun.password)
SessionsUUID tokens, 7-day expiry, stored in _sessions table
Response shape{ data, error } — Supabase v2 compatible throughout
Default port54321 (same as Supabase local dev)
LicenseMIT

Installation

Run with Bun (recommended)

# Run directly without installing
bunx busybase serve

# Or install globally
bun install -g busybase
busybase serve

Download a standalone binary

Pre-built binaries are available for every release — no Bun or Node.js required:

# Linux x64
curl -L https://github.com/AnEntrypoint/busybase/releases/latest/download/busybase-linux-x64 -o busybase
chmod +x busybase && ./busybase serve

# macOS ARM64
curl -L https://github.com/AnEntrypoint/busybase/releases/latest/download/busybase-macos-arm64 -o busybase

Quick Start

# 1. Start the server
bunx busybase serve
# → BusyBase: http://localhost:54321
// 2. Connect and use (SDK)
import BB from "busybase";

const db = BB("http://localhost:54321", "local");

// Auto sign-in with keypair (anonymous, no email needed)
const { data: { user, session } } = await db.auth.keypair.signIn();

// Insert data (table created automatically)
await db.from("todos").insert({ title: "Learn BusyBase", done: false });

// Query
const { data: todos } = await db.from("todos").select("*").eq("done", false);
console.log(todos); // [{ id: "...", title: "Learn BusyBase", done: "false" }]

# 3. Test everything works
bunx busybase test
# → 50+ passed, 0 failed

REST API

BusyBase exposes a PostgREST-compatible REST API under /rest/v1/. All responses use the { data, error } shape.

PATCH /rest/v1/:table?eq.col=val Update rows matching filter
PUT /rest/v1/:table?eq.col=val Update rows (alias for PATCH)

# Response body: { data: [...], error: null, count: 42 }

Filters & Operators

Filters are applied as query parameters. Multiple filters are ANDed together. The or parameter creates OR clauses within a request.

# Single filter
GET /rest/v1/users?eq.role=admin

# Multiple filters (AND)
GET /rest/v1/products?gt.price=10<e.price=100&eq.active=true

# OR filter
GET /rest/v1/users?or=role.eq.admin,role.eq.moderator

# IN filter
GET /rest/v1/orders?in.status=pending,processing,shipped

# NOT filter
GET /rest/v1/users?not.role.eq=banned

# LIKE filter (case-sensitive, matches substring)
GET /rest/v1/users?like.name=ali
# matches "Alice", "Alistair", etc.

# IS NULL
GET /rest/v1/posts?is.deleted_at=null

Keypair Auth (Ed25519)

BusyBase's primary auth flow is anonymous-first keypair auth using Ed25519 via the WebCrypto API. Users get a persistent cryptographic identity without any sign-up form.

How it works
On first call, BusyBase generates an Ed25519 keypair in the browser/runtime, stores it in localStorage (browser) or memory (server), signs a server-issued nonce, and receives a session. The same keypair always maps to the same user ID — across devices, after restoring from backup.

Flow diagram

  1. Client calls GET /auth/v1/keypair → receives a one-time nonce (60s TTL)
  2. Client signs the nonce with its Ed25519 private key
  3. Client posts { pubkey, nonce, signature } to POST /auth/v1/keypair
  4. Server verifies signature with crypto.subtle.verify("Ed25519", ...)
  5. Server finds or creates user by pubkey, issues 7-day session
  6. User can later add an email/password to their account
import BB from "busybase";
const db = BB("http://localhost:54321", "local");

// ── Anonymous sign-in (auto-generates keypair on first call) ──
const { data, error } = await db.auth.keypair.signIn();
console.log(data.user.id);          // persistent UUID
console.log(data.session.access_token);

// ── Backup your keypair ──
const keys = db.auth.keypair.export();
// Save keys.privkey + keys.pubkey somewhere safe

// ── Restore on a new device ──
const { data: restored } = await db.auth.keypair.restore(
  keys.privkey,
  keys.pubkey
);
// restored.user.id === original user.id ✓

// ── Generate a fresh keypair manually ──
const kp = await db.auth.keypair.generate();
// kp.privkey, kp.pubkey (base64)

// ── Remove local keys (soft logout — can restore later) ──
db.auth.keypair.forget();

Upgrading to email/password

After signing in with a keypair, users can add an email and password to their account without losing their data:

// Must be signed in with keypair first
await db.auth.updateUser({
  email: "user@example.com",
  password: "securepassword",
  data: { name: "Alice" }
});
// Same user.id, now has email too

Email / Password Auth

// ── Sign up ──
const { data, error } = await db.auth.signUp({
  email: "user@example.com",
  password: "mypassword",
  options: {
    data: { name: "Alice", plan: "free" }  // stored as user_metadata
  }
});
// data.user.id, data.user.email, data.user.user_metadata

// ── Sign in ──
const { data: { user, session } } = await db.auth.signInWithPassword({
  email: "user@example.com",
  password: "mypassword"
});
// session.access_token — use in Authorization header
// session.expires_in === 604800 (7 days)

// ── Get current user ──
const { data: { user } } = await db.auth.getUser();

// ── Update user ──
await db.auth.updateUser({
  email: "new@example.com",   // triggers onEmailChange hook
  password: "newpassword",
  data: { name: "Alice B." }  // merged into user_metadata
});

// ── Sign out ──
await db.auth.signOut();  // invalidates session token on server

Sessions

Sessions are UUID tokens stored in the _sessions table. They expire after 7 days.

// ── Get current session (local, no network call) ──
const { data: { session } } = await db.auth.getSession();
// session.access_token
// session.refresh_token
// session.expires_in  === 604800
// session.expires_at  === Unix timestamp
// session.token_type  === "bearer"

// ── Auth state listener ──
const { data: { subscription } } = db.auth.onAuthStateChange((event, session) => {
  console.log(event);  // "INITIAL_SESSION" | "SIGNED_IN" | "SIGNED_OUT" | "USER_UPDATED"
  console.log(session?.access_token);
});

// Unsubscribe
subscription.unsubscribe();

// ── Manually set session (e.g. from server-side rendering) ──
await db.auth.setSession({
  access_token: "...",
  refresh_token: "..."
});
Authorization header
All REST API requests requiring auth must include Authorization: Bearer <access_token>. The SDK sets this automatically after sign-in.

Password Reset

BusyBase supports email-based password reset. A signed reset token is sent via SMTP (or your custom sendEmail hook). Reset tokens expire in 1 hour.

# 1. Request reset (POST /auth/v1/recover)
curl -X POST http://localhost:54321/auth/v1/recover \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com"}'
# Always returns { data: {}, error: null } (no email leak)
# 2. User clicks link: GET /auth/v1/verify?token=<TOKEN>&type=recovery
# On your page, collect new password, then:

# 3. Verify token + set new password (POST /auth/v1/verify)
curl -X POST http://localhost:54321/auth/v1/verify \
  -H "Content-Type: application/json" \
  -d '{"token":"<TOKEN>","type":"recovery","password":"newpassword"}'
# Returns { data: { user, session }, error: null }
Custom reset email
Use the onPasswordReset hook to customize the reset email, or the sendEmail hook to use a different email provider entirely.

Realtime (WebSocket)

BusyBase supports real-time subscriptions over WebSocket, compatible with the Supabase channel() API pattern. Clients receive INSERT, UPDATE, and DELETE events as they happen.

Supabase-compatible
The SDK's channel().on().subscribe() API mirrors Supabase's realtime interface. Switch your URL and existing realtime code works.

WebSocket endpoint

Connect to ws://localhost:54321/realtime/v1/websocket. Send JSON messages to subscribe/unsubscribe:

// Raw WebSocket usage
const ws = new WebSocket("ws://localhost:54321/realtime/v1/websocket");

// Subscribe to a table
ws.send(JSON.stringify({ type: "subscribe", table: "todos" }));

// Receive events
ws.onmessage = (e) => {
  const msg = JSON.parse(e.data);
  // msg.eventType: "INSERT" | "UPDATE" | "DELETE"
  // msg.table: "todos"
  // msg.new: { id, title, done, ... } | null
  // msg.old: { id, title, done, ... } | null
  console.log(msg.eventType, msg.new, msg.old);
};

// Unsubscribe
ws.send(JSON.stringify({ type: "unsubscribe", table: "todos" }));

SDK channel API

import { createClient } from "busybase";
const db = createClient("http://localhost:54321", "local");

// Subscribe to all changes on "todos"
const channel = db.channel("my-channel")
  .on("postgres_changes", { event: "*", schema: "public", table: "todos" },
    (payload) => {
      console.log(payload.eventType); // INSERT, UPDATE, or DELETE
      console.log(payload.new);       // new row (null on DELETE)
      console.log(payload.old);       // old row (null on INSERT)
    })
  .subscribe((status) => {
    console.log("Channel status:", status); // "SUBSCRIBED"
  });

// Filter by event type
db.channel("inserts-only")
  .on("postgres_changes", { event: "INSERT", schema: "public", table: "todos" },
    (payload) => console.log("New todo:", payload.new))
  .subscribe();

// Cleanup
channel.unsubscribe();
db.removeAllChannels();

Embedded Mode

BusyBase can run entirely in-process with no HTTP server. Use createEmbedded() for zero-config local usage — same API as the HTTP SDK but operating directly on a local SQLite file via libSQL.

Setup

import { createEmbedded } from "busybase/embedded";

const db = await createEmbedded({ dir: "./my-data" });

// Same API as the HTTP client:
await db.from("todos").insert({ title: "Buy milk", done: false });
const { data } = await db.from("todos").select("*").eq("done", false);

// Auth works too:
await db.auth.signUp({ email: "user@test.com", password: "pass123" });
const { data: session } = await db.auth.signInWithPassword({
  email: "user@test.com", password: "pass123"
});

Configuration

interface EmbeddedConfig {
  dir?: string;    // Data directory (default: "busybase_data") — libsql backend only
  hooks?: Hooks;   // Hook functions (same interface as BUSYBASE_HOOKS file)
  backend?: string; // Registered backend name (default: "libsql")
  url?: string;    // Override connection URL passed to backend factory
}

Pluggable backends

The default backend is libSQL. Register other backends via registerBackend(name, factory) — the factory returns any object implementing @libsql/client's Client interface. This lets busybase run on sql.js, remote sqlite-over-HTTP, or any custom storage exposing execute({sql, args}).

import { registerBackend, createEmbedded } from "busybase/embedded";

// Example: sql.js in browser (zero server, zero native deps)
registerBackend("sqljs", async ({ url }) => {
  const { createClient } = await import("./libsql-sqljs.js");
  return createClient({ url });
});

const db = await createEmbedded({ backend: "sqljs", url: "file:appdb" });
await db.from("notes").insert({ body: "hello from a browser tab" });

The thebird browser-native web-OS uses this to host busybase entirely in-browser — each instance gets its own IndexedDB-backed sqlite blob via vendored sql.js. No server, full Supabase JS v2 surface.

Unknown backend names throw busybase: unknown backend '<name>'. mkdirSync only runs for the libsql backend so non-disk backends work cleanly.

Realtime in embedded mode

Realtime events fire via Node.js EventEmitter instead of WebSocket:

const db = await createEmbedded();

db.channel("my-channel")
  .on("postgres_changes", { event: "*", schema: "public", table: "todos" },
    (payload) => console.log(payload.eventType, payload.new))
  .subscribe((status) => console.log(status)); // "SUBSCRIBED"

await db.from("todos").insert({ title: "Test" });
// Logs: INSERT { id: "...", title: "Test" }

Hooks in embedded mode

import type { Hooks } from "busybase/hooks";

const hooks: Hooks = {
  beforeInsert: (table, rows) => {
    if (table === "protected") return { error: "Cannot insert into protected table" };
  },
  canAccess: ({ user, table, method }) => {
    if (!user && table !== "public") return { error: "Auth required" };
  }
};

const db = await createEmbedded({ hooks });
When to use embedded mode
Use embedded mode for CLI tools, desktop apps, serverless functions, or any context where running a separate HTTP server is unnecessary. The same data directory can be shared between embedded and server modes (but not simultaneously).

Hooks System

BusyBase loads a hooks file at startup. Point BUSYBASE_HOOKS to a TypeScript (or JS) file that exports hook functions. Every hook is optional.

BUSYBASE_HOOKS=./hooks.ts bunx busybase serve
Return values
  • Return nothing / void → operation proceeds normally
  • Return { error: "message" } or false → operation aborted, error returned to client
  • Return a value from pipe hooks (beforeSelect, afterSelect, afterInsert, afterUpdate) → value replaces the original rows/params
  • Throw an error → error message returned to client

  if (!user && method !== "GET")
    return { error: "Authentication required" };

  // Only admins can access _admin tables
  if (table.startsWith("_admin") && user?.role !== "admin")
    return { error: "Admin access required" };

  // Users can only access their own data
  // (combine with beforeSelect to add user_id filter)
};

export const beforeSelect = (table, params) => {
  // Inject user_id filter for user-owned tables
  const ownedTables = ["notes", "todos", "files"];
  if (ownedTables.includes(table) && currentUserId) {
    return { ...params, "eq.user_id": currentUserId };
  }
};

Pipe hook — called after rows are fetched. Return modified rows to filter/transform what the client receives.
export const afterSelect = (table, rows) => {
  if (table === "products") {
    // Remove internal cost fields
    return rows.map(({ internal_cost, ...r }) => r);
  }
};

Email / SMTP

BusyBase includes a built-in SMTP client (using Bun TCP, zero dependencies) for sending password reset emails. Configure it with environment variables.

BUSYBASE_SMTP_HOST=smtp.resend.com
BUSYBASE_SMTP_PORT=587
BUSYBASE_SMTP_USER=resend
BUSYBASE_SMTP_PASS=re_your_secret_key
BUSYBASE_SMTP_FROM=noreply@myapp.com
Prefer a third-party provider?
Use the sendEmail hook to route emails through Resend, SendGrid, AWS SES, Postmark, or any other provider. The hook completely replaces the SMTP transport.

If no SMTP is configured and no sendEmail hook is set, BusyBase logs the email to stdout:

[BusyBase] No email transport configured. Would send to user@example.com: Reset your password

SDK Reference

The BusyBase SDK mirrors the Supabase JS client v2 API. Import and initialize:

import BB from "busybase";
// or: import { createClient } from "busybase";
// or: const BB = require("busybase");

const db = BB("http://localhost:54321", "your-api-key");
// db.from(table)
// db.auth

All methods return Promise<{ data: T | null, error: Error | null }>.

auth.*

auth.keypair.signIn(privkeyB64?)

Sign in anonymously with a keypair. On first call (no stored keypair), generates a new Ed25519 keypair, persists it in localStorage (browser) or memory (server), and creates a new user. Subsequent calls reuse the stored keypair.

Pass a privkeyB64 string to sign in with a specific private key (must have been previously stored with its pubkey).

auth.keypair.restore(privkey, pubkey)

Restore from a saved backup. Sets the keypair in storage and calls signIn. Returns the same user as the original keypair.

auth.keypair.generate()

Generate a new Ed25519 keypair without signing in. Returns { privkey: string, pubkey: string } (base64 encoded).

auth.keypair.export()

Returns the current keypair from storage: { privkey: string | null, pubkey: string | null }. Save these for account recovery.

auth.keypair.forget()

Remove the keypair from local storage. The account still exists on the server — restore it later with auth.keypair.restore().

auth.signUp({ email, password, options? })

Register a new user with email + password. options.data is stored as user_metadata. Returns { data: { user, session: null }, error } — no auto sign-in.

auth.signInWithPassword({ email, password })

Sign in with email + password. Returns { data: { user, session }, error }.

auth.signOut()

Sign out and invalidate the current session token on the server.

auth.getUser()

Fetch the current user from the server (requires valid session token). Returns { data: { user }, error }.

auth.getSession()

Return the current in-memory session object (no network call). Returns { data: { session }, error: null }.

auth.updateUser({ email?, password?, data? })

Update the current user's email, password, or metadata. data is merged (not replaced) into user_metadata.

auth.setSession({ access_token, refresh_token })

Manually set the session (e.g. from SSR). The SDK will use this token for all subsequent requests.

auth.onAuthStateChange(callback)

Subscribe to auth state changes. The callback fires immediately with INITIAL_SESSION, then on sign-in, sign-out, and user updates. Returns { data: { subscription: { unsubscribe() } } }.

const { data: { subscription } } = db.auth.onAuthStateChange((event, session) => {
  // event: "INITIAL_SESSION" | "SIGNED_IN" | "SIGNED_OUT" | "USER_UPDATED"
});
subscription.unsubscribe();

from().*

from(table).select(columns?)

Start a SELECT query. columns is a comma-separated string (default: "*"). Chain filters and modifiers, then await to execute.

const { data } = await db.from("todos").select("id,title,done");

from(table).insert(data)

Insert one or more rows. data can be an object or array. id is auto-generated as a UUID if not provided. Tables are created automatically on first insert.

const { data } = await db.from("todos").insert({ title: "Buy milk" });
await db.from("todos").insert([{ title: "A" }, { title: "B" }]);

from(table).update(data)

Update rows matching a filter. Must chain at least one filter method. Returns updated rows.

const { data } = await db.from("todos").update({ done: true }).eq("id", "abc");

from(table).upsert(data)

Alias for insert in BusyBase. Inserts new rows (no true SQL UPSERT).

from(table).delete()

Delete rows matching a filter. Must chain at least one filter method.

await db.from("todos").delete().eq("done", true);

Chain filter methods onto any query. Multiple filters are ANDed together.

// Pagination example
const PAGE_SIZE = 20;
const page = 2;
const { data, count } = await db.from("posts")
  .select("id,title,created_at")
  .eq("published", true)
  .order("created_at", { ascending: false })
  .range(page * PAGE_SIZE, (page + 1) * PAGE_SIZE - 1)
  .count("exact");

Row-level security with hooks

// hooks.ts
let currentUser: any = null;

export const onRequest = async (req) => {
  // Parse user from token for use in other hooks
  const token = req.headers.get("Authorization")?.split(" ")[1];
  if (token) currentUser = await getUserFromToken(token);
  else currentUser = null;
};

export const canAccess = ({ user, table, method }) => {
  if (!user && method !== "GET")
    return { error: "Authentication required" };
};

export const beforeSelect = (table, params) => {
  // Users can only see their own rows
  const ownedTables = ["todos", "notes", "files"];
  if (ownedTables.includes(table) && currentUser) {
    return { ...params, "eq.user_id": currentUser.id };
  }
};

export const beforeInsert = (table, rows) => {
  // Auto-set user_id on insert
  const ownedTables = ["todos", "notes", "files"];
  if (ownedTables.includes(table) && currentUser) {
    rows.forEach(r => r.user_id = currentUser.id);
  }
};

Express-style middleware with onRequest

// hooks.ts
const rateLimit = new Map<string, number[]>();

export const onRequest = async (req) => {
  const ip = req.headers.get("x-forwarded-for") || "unknown";
  const now = Date.now();
  const window = 60_000; // 1 minute
  const limit = 100;     // requests per minute

  const times = (rateLimit.get(ip) || []).filter(t => now - t < window);
  times.push(now);
  rateLimit.set(ip, times);

  if (times.length > limit) {
    return new Response(
      JSON.stringify({ data: null, error: { message: "Rate limit exceeded" } }),
      { status: 429, headers: { "Content-Type": "application/json" } }
    );
  }
};
